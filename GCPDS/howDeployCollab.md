# GCPDS Deployment & Collaboration Guide {#howDeployCollab}

This guide explains how to contribute to the GCPDS environment documentation and ensure changes are correctly deployed to the personal domain.

## Workflow Overview

To maintain a professional and consistent documentation flow, follow these steps:

1.  **Modify Content**: Edit the Markdown (`.md`) files within the `GCPDS/` directory. Ensure you use unique IDs for pages using the `{#page_id}` syntax and link them using `@subpage`.
2.  **Regenerate & Deploy**: Use the provided automation script to rebuild the Doxygen HTML and push changes to the relevant branches.

## Using the Deployment Script

We have provided a PowerShell script `deploy_docs.ps1` in the root directory that automates the entire process.

### Prerequisites
- **Doxygen**: Must be installed and available in your PATH.
- **Git**: Configured with access to the remote repository.

### Execution
Run the following command from the project root:

```powershell
.\deploy_docs.ps1
```

### What the Script Does:
1.  **Regenerates Documentation**: Runs Doxygen using `GCPDS/agentCLI/Doxyfile` and outputs the HTML to the `docs/` folder.
2.  **Stages Changes**: Automatically adds all modified Markdown files and the generated HTML.
3.  **Commits**: Creates a commit with an automated timestamp.
4.  **Deploys**: Pushes the changes to the `main` branch (for source control) and the `gh-newpage` branch (for personal domain deployment).

## Professional Standards
- **No Empty Pages**: Ensure every link in the Doxygen navigation tree points to a file with actual content.
- **Unique IDs**: Always use unique, lowercase IDs for your pages to avoid routing collisions.
- **Commit Messages**: The script uses a generic message; for significant manual changes, consider manual commits with descriptive messages.
